## What's verified on Mac dev (this session)

These ran green on macOS 25.3.0 with Python 3.12.7 and no Photoshop /
pywin32 / Inno Setup installed:

- **`pytest tests/ -q` — 125 passed, 1 skipped** (the 1 skipped is a
  Windows-only branch of `_default_log_dir`).
- **`py_compile`** clean on all `.py` files.
- **End-to-end logging smoke** — fake module log calls land in
  `~/.tafpsd/logs/app.log` with module names + timestamps.
- **End-to-end persistence smoke** — synthesized v1 file migrates
  transparently to v2 on next save; corrupt file gets quarantined to
  `.broken.<ts>.json`.
- **End-to-end updater smoke** — UNC nonexistent path returns None
  within 0.5 s; hung worker @ 10 s with 2 s deadline aborts at 2.00 s
  (deadline honored, daemon thread leaks per design).
- **End-to-end ps_driver watchdog** — mocked pywin32 simulating 10 s
  hung COM call + 2 s deadline raises `TimeoutError` at 2.00 s with
  the actionable Chinese hint.
- **End-to-end version pipeline** — `_version.py` edit → 9.9.9 →
  `tools/render_version.py` → `version_info.txt` shows
  `(9, 9, 9, 0)` + `u'9.9.9'`; `installer.iss` reads `GetEnv` env;
  three-artifact alignment confirmed in one script call.
- **End-to-end NAS unreachable flow** — first launch enqueues toast,
  user clicks "不再提醒" → muted_warnings persists → second launch
  silent.
- **Build pipeline simulation** (steps 4/5/6/7/8 — skipping
  Windows-only PyInstaller + ISCC) — all green.
- **Cross-file consistency tests** — `APP_MUTEX_NAME` ↔
  `installer.iss MutexName` ↔ no spaces; `launcher.APP_VERSION` ↔
  `_version.__version__`; `installer.iss` uses `GetEnv` + `#error`
  fail-fast.

## What must be verified on Windows (alpha session)

The five things below need real toolchain access (Windows + PS + pywin32
+ Inno Setup 6) and can't be exercised on Mac.

### 1. Full `build.bat` produces a clean installer

Smoke goal: a single invocation of `app\build.bat` on a fresh Windows
box produces `dist\installer\TA-F-PSD-DiffBatch-Setup-1.5.0.exe` and
exits zero.

- [ ] Step 1 (clean) wipes `build\` + `dist\`.
- [ ] Step 2 (deps from lock) installs without errors.
- [ ] Step 3 (ruff) reports clean — first ruff run may surface
      pre-existing style issues; if so, gate this step behind a soft
      fail for v1.5.0 and file a v1.5.1 task.
- [ ] Step 4 (pytest) lands 125 passing on Windows.
- [ ] Step 5 (render) writes `version_info.txt` with
      `(1, 5, 0, 0)` + `u'1.5.0'`.
- [ ] Step 6 (PyInstaller) — verify `console=False`, `upx=False`,
      excludes ML libs, `_internal\` contains `nas_config.example.json`
      (and `nas_config.json` if you set one up).
- [ ] Step 7 (ISCC) — verify `APP_VERSION` env is set before ISCC
      runs; `installer.iss #error` is NOT triggered.
- [ ] Step 8 (smoke) — exe launches and stays alive 3 s.

### 2. The 15-item GUI smoke checklist (production exe)

After `build.bat` succeeds, install the produced setup.exe on a clean
VM (or a colleague's machine). Run through each step; mark ✓ / ✗.

### 3. Upgrade path from 1.4.1

Critical because of the AppMutex rename — see CHANGELOG.md migration
notes.

- [ ] Install 1.4.1 (existing release). Launch it. Save a preset.
      Quit cleanly.
- [ ] Install 1.5.0 over the top.
- [ ] Launch 1.5.0.
  - Preset survives (transparent v1 → v2 schema migration).
  - About shows 1.5.0.
  - `.tafpsd_presets.json` now has `schema_version: 2` field.
- [ ] **Edge case**: install 1.5.0 while 1.4.1 is *still running*.
      Inno may not detect 1.4.1 (mutex name changed). Either close
      1.4.1 manually first, or accept Inno's "file in use" dialog;
      it'll surface and let the user decide. Document outcome.

### 4. Auto-update happy path (requires a 1.5.1 stand-in)

To exercise the update flow without burning a real version bump:

- [ ] Build 1.5.0 normally.
- [ ] Make a one-line edit (e.g. an About string), bump `_version.py`
      to `1.5.0.1`, build again, upload to a *test* NAS folder.
- [ ] Update test `latest.json` to point at the 1.5.0.1 installer.
- [ ] Launch the 1.5.0 build, point env `PSBATCH_UPDATE_URL` at the
      test manifest, observe:
  - dialog appears
  - UPDATE → progress → installer launches → app restarts → About
    shows 1.5.0.1
- [ ] Repeat with `MUTE THIS VERSION` instead of UPDATE — second
      launch must stay silent.

### 5. SmartScreen / Defender behavior

- [ ] Download the produced setup.exe from a browser on a clean
      Windows install (no exception added). Record whether
      SmartScreen blocks it. Document in
      `TROUBLESHOOTING.md` if behavior differs from what's there.

## Alpha dogfood checklist (the colleague's part)

Hand the installer to one colleague (decision D4 — must be named
before flipping RC → release). They run:

1. Install via the setup.exe. Note any SmartScreen / antivirus prompts
   and what they did to get past them.
2. Launch. Within 3 s the main window must appear, even if disconnected
   from the network.
3. Run one real packshot batch (≥ 5 files). Outputs land in
   `_normalized/`; originals untouched.
4. Close. Relaunch. Same UI, presets/scripts persist.
5. Disconnect from VPN/office network. Launch. The "NAS unreachable"
   toast should appear once. Click "不再提醒".
6. Reconnect. Maintainer publishes a 1.5.0.1 to the test NAS. Colleague's
   next launch should pop the update dialog. Walk through UPDATE all
   the way to the app restarting and showing 1.5.0.1.

After step 6 succeeds: 1.5.0 is **shipped**. Update
`releases/1.5.0/CHANGELOG.md`, update top-level
`CHANGELOG.md` (move [Unreleased] → [1.5.0]), push to NAS, announce.

## What's deferred to v1.5.1

Per `plan/2026-05-19-v1.5-optimization-plan.md §15`, the following are
intentionally **not** in this RC:

- Physical split of `launcher.py` → `ui/` submodules. The region
  banners in launcher (W3 D5.1) make the split mechanical when it
  happens.
- Auto-update failure UI fallback (manual setup.exe link in the
  error dialog).
- Shared `team_scripts/` library on NAS.
- ChineseSimplified.isl bundled in installer.iss.
- RUN BATCH structural transform (the original v1.4 main feature —
  rolled forward).
- perf_check.py performance smoke.
- Code signing / EV certificate (decision D8: not doing).
- GitHub Actions CI (decision D11: not doing).
